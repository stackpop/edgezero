# Plan: Replace Spin Seed Handler with Per-Backend Writers

**Status:** v1 — drafted 2026-06-04 in response to PR-thread security
concern that exposing `/__edgezero/config/seed` on every deployed Spin
app creates a permanent internet-facing attack surface owned by
EdgeZero, even with the Pass 1-7 hardening we just landed (16-byte
constant-time token, 256 KiB body cap, 1000-entry/64 KiB caps,
fail-closed token-first ordering).

**Goal:** Get `config push --adapter spin` off our embedded HTTP
endpoint and onto the runtime backend's own protocol — matching the
pattern Cloudflare (`wrangler kv bulk put`) and Fastly
(`fastly config-store-entry create`) already use. Delete the seed
handler from prod entirely; the only writers are (a) Spin's own SQLite
backend file for local dev and (b) `spin cloud key-value set` for
Fermyon-hosted prod.

## Why this PR, not a follow-up

The seed handler is currently default-on in the scaffold
(`run_app_with_seeder`). Every project generated by `edgezero new
--adapter spin` would ship the endpoint to prod. We can't ship that
default and clean it up later; the right move is to land the
deletion + replacement in the same PR that introduced the migration.

## Design

### Architecture (current → target)

```
BEFORE (Pass 1-7):
  config push --adapter spin
    └─> HTTP POST https://<deployed-app>/__edgezero/config/seed
          └─> run_app_with_seeder intercepts before app router
                └─> seed::SpinKvSeedWriter.set(label, key, value)
                      └─> spin_sdk::key_value::Store::set (inside wasm)

AFTER (this plan):
  config push --adapter spin
    └─> parse runtime-config.toml next to spin.toml
    └─> dispatch on backend type:
          ┌─ type = "spin"  → rusqlite-direct write to .spin/sqlite_key_value.db
          ├─ Fermyon Cloud  → shell `spin cloud key-value set` per entry
          │  (auto-detected from `[adapters.spin.commands].deploy`
          │   containing `spin deploy` or `spin cloud deploy`)
          ├─ type = "redis" → error: "use `redis-cli SET` directly"
          └─ type = "azure" → error: "use `az cosmosdb` directly"
```

Per-backend writers mirror what Cloudflare (Wrangler) and Fastly
(Fastly CLI) already do. No internet-facing endpoint owned by us; no
embedded HTTP write surface in the deployed wasm component; no token
rotation; no `--seed-url`/`--seed-token` chain to defend.

### SQLite-direct writer for `type = "spin"`

Spin's `key-value-spin` crate
([`crates/key-value-spin/src/store.rs`](https://github.com/spinframework/spin/blob/main/crates/key-value-spin/src/store.rs))
uses one table with this exact schema:

```sql
CREATE TABLE IF NOT EXISTS spin_key_value (
    store TEXT NOT NULL,
    key   TEXT NOT NULL,
    value BLOB NOT NULL,
    PRIMARY KEY (store, key)
)
```

Spin's `SET` statement is:

```sql
INSERT INTO spin_key_value (store, key, value) VALUES ($1, $2, $3)
ON CONFLICT(store, key) DO UPDATE SET value=$3
```

Our writer uses the SAME `CREATE TABLE IF NOT EXISTS` and the SAME
`INSERT … ON CONFLICT` statement. The file lives at
`<spin.toml dir>/.spin/sqlite_key_value.db` by default (Spin's
hard-coded default for `DatabaseLocation::Path`); operators can
override per-label via `[key_value_store.<label>] path = "..."` in
`runtime-config.toml`, which our parser must honour.

WAL mode (Spin's default) allows our writes to coexist with a running
`spin up` — both processes use shared SQLite locking. No need to stop
the app before pushing.

### Schema-drift protection

We vendor Spin's exact `CREATE TABLE` statement string as a constant
in our writer, with a comment naming the source file + git SHA we
pulled it from. A unit test asserts byte-equality between our string
and what `rusqlite` produces from our schema (catches lexical drift).
We pin `spin-sdk = "~6.0"` so any Spin breaking change is a build
failure (already in Cargo.toml as of Pass 8a).

If Spin changes the schema in a `6.1.x` release, the version pin
catches it as a build failure when the operator upgrades. The vendored
string + byte-compare test catches drift in our copy without touching
spin-sdk.

### Fermyon Cloud writer

Detection: `[adapters.spin.commands].deploy` contains `spin deploy` or
`spin cloud deploy`. If detected, push shells per entry:

```bash
spin cloud key-value set --store <platform-label> <key> <value>
```

The Fermyon Cloud plugin (`spin cloud`) handles authentication via
`spin cloud login`; we don't manage credentials. Pattern mirrors
`wrangler kv bulk put` (Cloudflare) and `fastly config-store-entry
create` (Fastly): operator authenticates the platform's CLI separately,
we shell out to it.

### CLI surface changes

**Deleted:**

- `--seed-url`, `--seed-token` flags on `ConfigPushArgs`.
- `EDGEZERO__ADAPTERS__SPIN__SEED_URL`,
  `EDGEZERO__ADAPTERS__SPIN__LOCAL_SEED_URL`,
  `EDGEZERO__ADAPTERS__SPIN__SEED_TOKEN` env vars.
- `[adapters.spin.commands].seed-url` manifest field.
- `AdapterPushContext::seed_url`, `seed_token` fields and the
  `with_seed_url`/`with_seed_token` builder methods.
- `Adapter::reserved_paths` trait method (only user was Spin's seed
  route).

**Added:**

- `--runtime-config <path>` flag on `ConfigPushArgs` (default:
  `runtime-config.toml` next to the spin manifest).

**Survivor (unchanged):**

- `--local` flag retains its general meaning across adapters
  (push to the local emulator). For Spin specifically `--local`
  becomes informational: the per-backend dispatch in `runtime-config.toml`
  already selects the local vs. cloud writer based on backend `type`;
  `--local` only suppresses the auto-Fermyon-Cloud detection so an
  operator can force the local SQLite path even when the deploy
  command is `spin deploy`.

### Runtime side: drop the seed handler

- Delete `crates/edgezero-adapter-spin/src/seed.rs` (entire file,
  ~700 lines including Pass 1-7 hardening).
- Delete `run_app_with_seeder` from
  `crates/edgezero-adapter-spin/src/lib.rs`.
- Revert
  `examples/app-demo/crates/app-demo-adapter-spin/src/lib.rs`
  to call `run_app::<App>` (drop the Pass 1 change).
- Revert
  `crates/edgezero-adapter-spin/src/templates/src/lib.rs.hbs`
  scaffold to call `run_app::<App>`.

### app-demo + scaffold runtime-config.toml

Stays in place. Spin still requires `[key_value_store.<label>]` to
declare custom KV labels — our writer reads this file too. Operators
no longer need anything else (no token, no URL chain).

  ```rust
      CREATE TABLE IF NOT EXISTS spin_key_value (
          store TEXT NOT NULL,
          key   TEXT NOT NULL,
  ```

  ```rust
  fn push_config_entries(...) -> Result<Vec<String>, String> {
      let runtime_config_path = push_ctx
          .runtime_config_path
          .map(Path::to_path_buf)
          .unwrap_or_else(|| {
              spin_manifest_dir(manifest_root, adapter_manifest_path)
                  .join("runtime-config.toml")
          });
      let parsed = runtime_config::parse(&runtime_config_path)?;
      let backend = parsed
          .key_value_stores
          .get(store.platform())
          .or(parsed.key_value_stores.get("default"));
      match (backend, fermyon_cloud_detected(manifest_root)) {
          (Some(KeyValueBackend::Spin { path }), _) =>
              push_sqlite::write_batch(...),
          (None | Some(KeyValueBackend::Spin { .. }), true) =>
              push_cloud::write_batch(...),
          (Some(KeyValueBackend::Redis { .. }), _) =>
              Err("redis backend not yet supported; ...")?,
          (Some(KeyValueBackend::AzureCosmos { .. }), _) =>
              Err("azure-cosmos backend not yet supported; ...")?,
          (Some(KeyValueBackend::Unknown { type_name, .. }), _) =>
              Err(format!("unknown backend type `{type_name}`"))?,
      }
  }
  ```
